# 봄봄 서비스 기본 구조

기획할 때 전제로 삼아야 하는 서비스 핵심 구조를 정리한다. product-planner 에이전트는 이 파일을 먼저 읽고 기획에 들어갈 것. 여기 적힌 전제는 사용자에게 다시 묻지 않는다.

> 출처: `docs/README.md` + 프로덕트 오너 인터뷰 (2026-04-19)

## 한 줄 정의

봄봄은 **"읽기의 라이프 로그"**. 단순한 뉴스레터 리더를 넘어, 읽는 습관을 만들고 읽은 콘텐츠를 개인 자산으로 바꿔주는 서비스다.

## 풀려는 문제

- 메일함에 개인·공적 메일, 여러 뉴스레터가 뒤섞여 **중요한 뉴스레터가 잊히거나 스팸으로 빠진다**.
- 구독만 해놓고 **습관이 없어 읽지 못하고 쌓이기만 한다**.
- 어디까지 읽었는지, 무엇이 중요했는지 **기록이 남지 않아 흘러간다**.

## 타겟층

1. 뉴스레터를 **편리하게 보고 싶은** 사람 (정리 효율)
2. 뉴스레터를 구독 중이지만 **습관이 없어 못 읽는** 사람 (습관 형성)
3. 뉴스레터에 관심은 있지만 **아직 시작하지 못한** 사람 (온보딩)

→ 새 기능 기획 시 **어느 타겟을 주로 겨냥하는지** 명시할 것.

## 봄봄의 가치 명제 (기능 판단 기준)

새 기능은 보통 아래 5가지 가치 중 하나에 기여한다. 기획 시 **어느 축인지** 짚을 것.

| # | 사용자 문제 | 봄봄의 해결 방식 | 전달 가치 |
|---|---|---|---|
| 1 | 뉴스레터가 쌓여만 간다 | 읽기 기록 + 알림 + 목표 설정 | **읽는 습관 형성** |
| 2 | 어디까지 읽었는지 기억 안 남 | 읽기 진행률 자동 기록 | **편의성** |
| 3 | 혼자 읽으면 재미 없다 | 팔로우 / 리더보드 비교 | **소셜 기반 동기 유발** |
| 4 | 뉴스레터 내용이 흘러간다 | 하이라이트 · 메모 저장 | **지식 자산화** |
| 5 | 각종 메일이 섞여 있다 | 통합 수신 + 필터링 | **콘텐츠 정리 효율성** |

## 핵심 기능 맵

> ⚠️ 이 섹션은 **계속 변경된다**. 새 기능이 출시되거나 기존 기능이 개편될 때마다 갱신. 갱신 규칙은 이 문서 하단 "이 문서 유지보수" 참고.

_마지막 업데이트: 2026-04-19_

| 기능 | 설명 | 연결되는 가치 |
|---|---|---|
| 뉴스레터 통합 수신 | 여러 이메일에서 오는 뉴스레터를 한 곳에 모음 | 5 |
| 읽기 트래킹 | 읽은 정도 · 시간 · 날짜 자동 기록 | 1, 2 |
| 읽기 목표 설정 | 일/주 단위 목표 설정, 진행률 시각화 | 1 |
| 하이라이트 · 메모 | 중요한 부분을 저장 | 4 |
| 팔로우 · 비교 | 다른 사용자의 읽기 현황 비교, 리더보드 | 3 |
| 아카이빙 | 메모를 지식 노트처럼 관리 | 4 |

→ 새 기능이 기존 기능과 **어떻게 연결되는지** (예: 매일메일 답변 기능 → 하이라이트·메모·아카이빙 연계 가능) 기획에서 검토할 것.

## 계정 모델

- **가입/로그인은 소셜 로그인 전용** (Google / Apple / Kakao 등). 자체 이메일·비밀번호 계정 없음.
- **비로그인 사용자는 서비스 본질 기능을 이용할 수 없다.** 구독·수신·열람·기록 모두 로그인 필수.
- 따라서 기획에서 "비로그인 처리"는 대부분 **로그인 유도 화면 유무**로 귀결된다. 로컬 임시 저장이나 게스트 제출 같은 분기는 기본적으로 고려 대상이 아니다.

## 구독 모델

- 회원가입 시 **봄봄이 전용 이메일 주소를 발급**한다 (예: `xxx@bombom.news`).
- 유저는 이 전용 이메일을 외부 뉴스레터(매일메일 등) 구독 폼에 입력해 구독한다.
- **구독 가능한 뉴스레터는 봄봄이 큐레이션한 리스트에 한정된다.** 유저가 임의 뉴스레터를 등록할 수 없다.
- → 뉴스레터별 특성(발송 주기, 포맷, 인터랙티브 요소)을 봄봄이 사전에 파악·관리할 수 있다. 뉴스레터별 맞춤 UI를 얹는 기획이 가능하다.

## 수신·열람 플로우

1. 유저가 봄봄 전용 이메일로 외부 뉴스레터 구독
2. 외부 뉴스레터 서비스가 해당 이메일로 메일 발송
3. 봄봄 서버가 메일을 수신·저장
4. 유저가 봄봄 앱/웹에서 메일을 열람

→ **이메일은 외부에서 유입되는 콘텐츠**다. 봄봄이 콘텐츠를 직접 생산·편집하지 않는다.

## 렌더링 방식

- **하이브리드**: 기본은 이메일 HTML 원본을 그대로 보여주고, 그 위에 봄봄 고유 UI(읽기 진행률, 하이라이트/메모 등)를 오버레이한다.
- 뉴스레터 원본에 있는 버튼·링크·이미지는 그대로 동작한다.
- 봄봄의 새 기능은 보통 **원본 위 오버레이** 또는 **원본 바깥 화면(리스트·기록·대시보드)**에서 구현된다. 원본 HTML 자체를 고치지 않는다.

## 주요 화면 구조

> ⚠️ 현재 스코프 기준. 신규 페이지 추가 시 갱신 필요.

_마지막 업데이트: 2026-04-19_

### 현재 존재
- **오늘의 뉴스레터**: 상단 탭(오늘 / 보관함) + 검색 + 필터(읽음/읽지않음). 메인에 오늘 받은 뉴스레터 카드(제목·종류·초반 메시지·카테고리·발행 시간·예상 읽는 시간). 사이드에 프로필·이메일(복사)·연속 읽기·오늘 진행률·주간 목표·빠른 실행.
- **뉴스레터 추천**: 카테고리별로 큐레이션된 뉴스레터 나열. 상단 배너.

### 추후 추가 예정
- 랜딩 페이지
- 로그인 페이지
- **뉴스레터 읽기 페이지** (← 매일메일 답변 기능 등 "열람 중" 인터랙션 기능들이 이 페이지에 붙음)

→ 새 기능 기획 시 **어느 화면에 붙는지** 명시할 것.

## 기획 시 반드시 전제로 삼을 것

- 유저는 **항상 로그인 상태**다. 비로그인 분기를 길게 논의하지 말 것.
- 뉴스레터 콘텐츠 원본은 외부 소유다. **원본 HTML을 변형하는 방향 기획은 지양**.
- 큐레이션된 뉴스레터에 한해 **뉴스레터별 특화 기능**(예: 매일메일 답변 기능)을 봄봄이 사전 지식으로 얹을 수 있다.
- 뉴스레터 열람은 봄봄 앱/웹에서 일어난다. **이메일 클라이언트(Gmail 등) 호환성은 고려 대상이 아니다.**
- **소셜·자산화는 봄봄의 DNA**다. 개인 기능으로 시작하는 기획이어도 "추후 팔로우·리더보드·아카이빙으로 확장 가능한가"를 고려사항에 남길 것.
